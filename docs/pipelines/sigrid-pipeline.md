# Sigrid CI Pipeline

This pipeline connects the Brain Wiz repository to Sigrid. Sigrid analyzes the source code and reports quality information such as maintainability, architecture, open source health, and security.

The Sigrid workflow is separate from the normal build, lint, format, and test pipeline. The existing CI checks whether the project still works. Sigrid checks whether the project remains maintainable over time.

---

## Overview

The Sigrid flow starts when code is pushed to one of the main project branches. GitHub Actions checks out the repository, uploads the source code to Sigrid, and stores the Sigrid output as a workflow artifact.

```text
Code is pushed to develop or master
        ↓
GitHub Actions starts the Sigrid workflow
        ↓
Repository is checked out
        ↓
Sigrid CI uploads the source code
        ↓
Sigrid analyzes the project
        ↓
Results become visible in the Sigrid dashboard
        ↓
Sigrid output is saved as a GitHub Actions artifact
```

The goal is to make code quality visible during development. Sigrid does not automatically fix code, but it helps the team find maintainability risks and decide what should be improved.

## Core files and configuration

| File / location                | Purpose                                                    |
| ------------------------------ | ---------------------------------------------------------- |
| `.github/workflows/sigrid.yml` | Defines the GitHub Actions workflow for running Sigrid.    |
| `SIGRID_CI_TOKEN`              | GitHub repository secret used to authenticate with Sigrid. |
| Sigrid dashboard               | Shows the analysis results for the Brain Wiz system.       |
| `sigrid-ci-output`             | Output folder generated by Sigrid during the workflow run. |
| GitHub Actions artifact        | Stores the Sigrid output for debugging and review.         |

## Workflow trigger

The workflow runs automatically when code is pushed to:

```text
develop
master
```

It can also be started manually from the GitHub Actions page because it includes:

```yml
workflow_dispatch:
```

This is useful when the team wants to re-run Sigrid without creating a new commit.

```yml
on:
  push:
    branches:
      - develop
      - master
  workflow_dispatch:
```

## Authentication

Sigrid needs an authentication token to upload code from GitHub Actions.

The token is stored as a GitHub Actions secret:

```text
SIGRID_CI_TOKEN
```

The token must never be committed to the repository.

To add or update the token:

```text
GitHub repository
        ↓
Settings
        ↓
Secrets and variables
        ↓
Actions
        ↓
New repository secret
```

Use the following values:

```text
Name: SIGRID_CI_TOKEN
Value: <Sigrid token>
```

The workflow reads the token here:

```yml
env:
  SIGRID_CI_TOKEN: ${{ secrets.SIGRID_CI_TOKEN }}
```

## Sigrid project configuration

The workflow uploads the code to the following Sigrid project:

```yml
customer: uvapse
system: brain-wiz
```

| Value       | Meaning                                    |
| ----------- | ------------------------------------------ |
| `uvapse`    | The Sigrid customer identifier.            |
| `brain-wiz` | The system name for the Brain Wiz project. |

The customer value comes from the Sigrid URL:

```text
https://sigrid-says.com/uvapse/...
```

After the first successful run, Sigrid automatically created the `brain-wiz` system and made the dashboard available.

## Current workflow

The Sigrid workflow is defined as follows:

```yml
name: Sigrid

on:
  push:
    branches:
      - develop
      - master
  workflow_dispatch:

concurrency:
  group: sigrid-${{ github.ref }}
  cancel-in-progress: true

jobs:
  sigrid:
    name: Run Sigrid
    runs-on: ubuntu-latest
    timeout-minutes: 20

    permissions:
      contents: read

    steps:
      - name: Check out repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Run Sigrid CI
        uses: Software-Improvement-Group/sigridci@main
        with:
          customer: uvapse
          system: brain-wiz
          source: .
          publish: true
          capability: maintainability,osh,security
          exclude: node_modules,dist,coverage,.git
          showupload: true
          out: sigrid-ci-output
        env:
          SIGRID_CI_TOKEN: ${{ secrets.SIGRID_CI_TOKEN }}

      - name: Save Sigrid output
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sigrid-ci-output
          path: sigrid-ci-output/**
          retention-days: 7
          if-no-files-found: ignore
```

---

## Repository checkout

The first step checks out the repository:

```yml
- name: Check out repository
  uses: actions/checkout@v4
  with:
    fetch-depth: 0
```

`fetch-depth: 0` is used so the full Git history is available during the Sigrid run. This gives Sigrid more complete project context.

---

## Running Sigrid CI

The workflow uses the official Sigrid GitHub Action:

```yml
- name: Run Sigrid CI
  uses: Software-Improvement-Group/sigridci@main
```

The source code is uploaded from the root of the repository:

```yml
source: .
```

The workflow uses:

```yml
publish: true
```

This means the workflow uploads the project to Sigrid and waits for analysis feedback.

---

## Enabled Sigrid capabilities

The workflow enables multiple Sigrid capabilities:

```yml
capability: maintainability,osh,security
```

| Capability        | Purpose                                                                |
| ----------------- | ---------------------------------------------------------------------- |
| `maintainability` | Checks code structure, duplication, complexity, size, and testability. |
| `osh`             | Checks open source health and dependency-related risks.                |
| `security`        | Checks security-related risks in the project and dependencies.         |

This gives the team a broader quality overview than maintainability alone.

---

## Excluded folders

The workflow excludes generated or irrelevant folders:

```yml
exclude: node_modules,dist,coverage,.git
```

These folders are not part of the source code written by the team and should not influence the Sigrid analysis.

| Folder         | Reason                                           |
| -------------- | ------------------------------------------------ |
| `node_modules` | Installed dependencies, not project source code. |
| `dist`         | Generated build output.                          |
| `coverage`     | Generated test coverage output.                  |
| `.git`         | Git internals.                                   |

During setup, `showupload: true` was used to verify that no unnecessary folders were uploaded.

```yml
showupload: true
```

This can stay enabled while the team wants detailed upload logs. It can also be removed later if the logs become too verbose.

---

## Sigrid output artifact

The workflow stores Sigrid output as a GitHub Actions artifact:

```yml
- name: Save Sigrid output
  if: always()
  uses: actions/upload-artifact@v4
```

The output is saved from:

```text
sigrid-ci-output
```

The artifact is kept for 7 days:

```yml
retention-days: 7
```

This makes it easier to debug the workflow or inspect the Sigrid output from GitHub Actions.

---

## Extra workflow settings

### Concurrency

```yml
concurrency:
  group: sigrid-${{ github.ref }}
  cancel-in-progress: true
```

This prevents multiple Sigrid runs from running at the same time on the same branch.

If a new commit is pushed while an older Sigrid run is still running, the older run is cancelled.

### Timeout

```yml
timeout-minutes: 20
```

This prevents the workflow from running forever if something goes wrong.

### Minimal permissions

```yml
permissions:
  contents: read
```

The workflow only needs read access to the repository. It does not need write access because it does not change files in the repository.

## How to test the workflow

To test the Sigrid workflow:

1. Make sure the `SIGRID_CI_TOKEN` secret exists in GitHub.
2. Push a commit to `develop` or `master`.
3. Open the GitHub repository.
4. Go to the **Actions** tab.
5. Open the **Sigrid** workflow.
6. Check that the workflow passes.
7. Open the Sigrid dashboard.
8. Confirm that the latest upload is visible.

The workflow can also be tested manually:

```text
GitHub Actions
        ↓
Sigrid workflow
        ↓
Run workflow
        ↓
Select branch
        ↓
Run
```

## What Sigrid does and does not do

Sigrid is a quality analysis tool. It helps the team understand where maintainability and quality risks exist.

Sigrid can help identify:

- complex code;
- duplicated code;
- large units;
- risky dependencies;
- security concerns;
- weaker components;
- maintainability trends over time.

Sigrid does not:

- run the test suite;
- build the application;
- format the code;
- lint the code;
- automatically refactor code;
- automatically fix maintainability issues;
- replace code review.

Those tasks are handled by other CI jobs and by the development team.

---

## How developers should use Sigrid

Developers should use Sigrid as a continuous quality monitor.

The recommended process is:

1. Check the dashboard after new uploads.
2. Use Code Explorer to find risky files or components.
3. Improve code quality while working on related features.
4. Keep refactorings small and focused.
5. Avoid adding unnecessary complexity or duplication.
6. Check whether quality improves after later uploads.

The goal is not to fix everything at once. The goal is to prevent the codebase from becoming harder to maintain over time.

---

## Relation to the normal CI pipeline

The Sigrid pipeline does not replace the existing CI checks.

```text
Build / lint / format / tests = verifies correctness
Sigrid = monitors long-term code quality
```

The normal CI pipeline checks whether the project still works. The Sigrid pipeline checks whether the project remains maintainable and healthy.

Both pipelines support the same goal: keeping Brain Wiz stable, readable, and easier to improve.
